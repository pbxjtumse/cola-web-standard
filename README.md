# 2. 建设背景

当前工程基于 COLA 分层架构进行组织，整体包含：

* `adapter`
* `app`
* `client`
* `domain`
* `infrastructure`
* `start`
* `component`

其中：

* `adapter / app / domain / infrastructure / start` 负责业务应用分层
* `component` 负责技术组件与基础设施能力沉淀

随着项目逐步演进，已经开始出现大量技术底座能力雏形，例如：

* 并发执行与线程池治理
* 并发隔离、限流、超时等治理能力
* 消息发送、消费、失败处理能力
* 事务增强能力
* 可观测性相关的日志、指标、MDC、trace 雏形

这些能力已经具备一定基础，但当前仍存在以下问题：

1. 组件边界尚未完全收敛
2. 零散能力尚未统一为标准化组件
3. 可观测性尚未真正成为公共底座
4. 聚合模块、可依赖模块的边界曾出现混用问题
5. 文档与代码沉淀仍需进一步系统化

因此，当前阶段核心目标为：

**在保留现有 COLA 主体分层结构的前提下，对技术组件体系进行系统化整理与分阶段重构。**

# 3. 总体建设目标

技术组件体系的长期目标如下：

## 3.1 统一基础能力

将缓存、消息、幂等、锁、治理、任务执行、可观测性、配置、事务等共性能力沉淀为统一组件，避免业务系统重复建设。

## 3.2 统一使用方式

每一个组件都应具备统一的：

* 配置方式
* 接入方式
* 扩展方式
* 监控方式
* 故障处理方式

## 3.3 统一治理能力

所有组件最终都需要纳入统一治理体系，包括：

* 日志
* 指标
* Trace / 上下文透传
* 异常分类
* 告警接入
* 性能与容量分析

## 3.4 保持实现可替换

技术组件不应与单一实现强绑定。

例如：

* 可观测性组件不绑定单一 APM 平台
* 消息组件不绑定单一 MQ 产品
* 数据源能力不绑定单一访问实现
* 幂等组件不绑定单一入口场景

## 3.5 面向后续技术中台建设

本组件体系不仅服务当前项目，还将作为后续技术中台、业务平台与复杂业务系统的公共底座。

# 5. 架构原则

## 5.1 业务与技术组件解耦

技术组件不依赖具体业务模型，不承载业务语义，不将业务逻辑沉淀到基础组件内部。

## 5.2 聚合模块与可依赖模块分离

在 Maven 多模块结构中：

* 聚合模块仅负责聚合与管理，必须为 `packaging = pom`
* 可依赖模块才是真正产出 jar 的组件模块

该原则已经在本轮重构与构建排错过程中被明确并验证。

## 5.3 先收敛边界，再补功能深度

当前阶段优先完成：

* 职责定义
* 模块边界
* 依赖关系
* 配置入口
* 可观测接入方式

而不是一开始把所有功能一次性堆满。

## 5.4 组件必须可观测

所有技术组件最终都必须接入统一观测体系，至少具备：

* 日志输出能力
* 指标埋点能力
* 异常统计能力
* 上下文透传能力

## 5.5 长期演进优先于短期功能堆叠

当前建设目标是形成稳定的长期底座，而不是短期堆积功能点。

# 8. 当前进行中的工作

## 8.1 可观测性组件作为第一优先级推进

当前已经明确：

**第一个正式进入系统化设计和落地阶段的技术组件为：可观测性组件。**

原因：

1. 它是所有技术组件的底座
2. 不先做观测，后续组件难以统一治理
3. 当前项目中已经零散存在日志、指标、MDC、trace 相关能力，适合统一抽取
4. Spring Boot 3 为 observability 提供了更清晰的官方演进路径

## 8.2 可观测性组件定位已明确

当前对可观测性组件的定义是：

> 面向项目内部的统一观测抽象层与接入层，而不是重写一套外部 APM 平台。

其定位包括：

* 统一日志规范
* 统一 metrics 规范
* 统一上下文透传
* 统一 traceId / requestId 模型
* 统一异常观测
* 统一组件级埋点方式

同时已明确：

* 它不应绑定单一 APM 实现
* 后续可对接 SkyWalking、OpenTelemetry、Prometheus/Grafana 等后端平台

## 8.3 Boot 3 升级后的运行时问题正在清理

虽然 Maven 构建已经成功，但运行时仍暴露出一些问题，例如：

* 数据源自动配置触发
* MySQL 驱动缺失导致 `DataSource` 初始化失败

当前已经明确：

* 这不是数据库连通性问题
* 而是运行时依赖与自动配置行为需要进一步梳理

这部分属于当前运行时兼容性清理范围。

## 8.4 数据访问默认采用连接池方案的方向已确定

针对数据库访问层，当前已形成明确结论：

* 中台类、长期运行服务默认采用连接池
* 不在业务代码中直接使用 `DriverManager.getConnection()`
* 统一采用 `DataSource + HikariCP` 作为标准方案

这为后续数据源治理与连接池可观测接入奠定了基础。

# 10. 当前关键技术决策

## 10.1 使用 Spring Boot 3 作为后续长期底座

当前已经决定：

* 不再继续以 Boot 2 作为长期版本基线
* 后续技术组件体系统一建立在 Boot 3 + Java 17 基础之上

## 10.2 可观测性不绑定单一平台

当前已明确：

* 不将内部可观测性组件设计成某一个外部 APM SDK 的包装层
* 而是设计为内部统一抽象层

## 10.3 数据库访问标准化为连接池模式

当前已明确：

* 长期运行服务统一采用连接池模式
* 标准方案为 `DataSource + HikariCP`
* 禁止业务代码直接手写直连数据库建连逻辑

## 10.4 技术组件按阶段推进

当前建设策略为：

### 第一批

* 可观测性组件
* 服务治理组件
* 任务与执行组件
